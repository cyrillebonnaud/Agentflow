# PRD Writing

## What
A Product Requirements Document (PRD) is the authoritative written description of what is being built, why, for whom, and how success will be measured. It is the contract between product, design, and engineering for a scope of work.

## How
1. Write a one-paragraph problem statement: who has this problem, why it matters, what happens if it goes unsolved
2. Define the target user and their relevant job-to-be-done (not demographics — behavior and motivation)
3. State success metrics: primary metric (the one number that matters most), secondary metrics, and guardrail metrics (things we must not break)
4. Define scope:
   - In scope: features and behaviors this document covers
   - Out of scope: explicit non-goals that prevent scope creep
5. Describe requirements as user-facing behaviors, not implementation details (use "users can…" not "the system will…")
6. List open questions with owners and due dates
7. List assumptions that, if wrong, would change the scope materially
8. Include a glossary for any domain terms that could be interpreted differently by different readers
